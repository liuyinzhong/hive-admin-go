# Hive Admin Go 后端开发规范

## 项目范围

`hive-admin-go` 是 Hive 系统后端服务，当前技术栈以 `go.mod` 和现有代码为准：

- Go `1.25.0`
- Gin
- GORM + MySQL
- JWT Bearer Token
- Swagger（已启用）

主要目录：

- `router`：路由注册和分组。
- `controllers`：HTTP 参数绑定、校验和响应。
- `services`：业务逻辑和数据库操作。
- `models`：数据库模型、请求/响应结构。
- `database`：数据库初始化和连接。
- `middleware`：认证等中间件。
- `utils`：项目公共工具。
- `docs`：Swagger 自动生成文件。

## 开发前

按以下顺序获取上下文：

1. 阅读当前模块的 Router、Controller、Service 和 Model。
2. 查找相似接口、分页、事务、导入导出和数据库操作。
3. 阅读 Swagger 注释和相关 SQL。
4. 核对前端调用方式和字段类型。
5. 确认没有成熟实现后再设计新方案。

项目代码是重要参考，但不要机械复制旧问题。现有代码、文档和规则不一致时，先说明差异。

## 技能自动调用

本仓库技能位于 `.agents/skills`，遵循工作区根 `AGENTS.md` 的通用技能路由。任务与技能说明匹配时，无需用户点名，必须先阅读对应 `SKILL.md` 再使用。

- 新增或修改 Router、Controller、Service、Model/DTO 时，使用 `implement`；Service、工具函数、参数校验或缺陷回归存在稳定测试边界时，同时使用 `tdd`，沿用 Go 原生测试和项目现有验证命令。
- 接口异常、事务问题、GORM 查询错误、性能回退或偶发失败的根因不明时，使用 `diagnosing-bugs`，先建立不启动服务、不连接生产环境的最小反馈命令。
- 设计分层、事务边界、查询接口、公共 Service 或适配器时，使用 `codebase-design`，并继续遵守 Router -> Controller -> Service/Model 的依赖方向。
- 前后端字段、DTO、数据库模型、枚举或业务术语存在歧义时，使用 `domain-modeling`；需要集中澄清并沉淀文档时组合 `grilling` 与 `grill-with-docs`。
- 只有纯业务状态或算法问题需要可运行实验时才使用 `prototype`；原型不得连接生产数据库、执行迁移或触发 Apifox 同步。
- 完成有实质代码变更的实现后使用 `code-review`，重点核对分层、统一响应、参数校验、事务、权限、SQL 安全、Swagger 和前后端契约。
- 只有用户明确要求规格、任务拆分或架构巡检时，才使用 `to-spec`、`to-tickets`、`improve-codebase-architecture`；不得自动向外部工单系统发布内容。
- 技能不能作为修改认证授权、数据库结构、公共响应、配置加载、依赖或生成文档流程的默认授权。

## 分层与依赖

允许的主要依赖方向：

```text
Router -> Controller
Controller -> Service、Model
Service -> Database、Model、Utils
Middleware -> 认证及基础设施
```

- Router 只负责路由、分组和中间件注册。
- Controller 只负责参数绑定、参数校验、调用 Service 和返回统一响应。
- Service 负责完整业务、数据库操作、事务、数据组装和转换。
- Model 定义数据结构和映射，不访问 Controller、Service 或 Database。
- Service 不返回 Gin HTTP Response，Controller 不直接访问数据库或拼装 SQL。
- 不跨层调用，不在 Controller 中拆散同一业务事务。

## Router 与接口

- 保持现有 `/api/auth`、`/api/system`、`/api/dev`、`/api/workflow` 等模块划分。
- 普通资源接口保持 RESTful 风格，路径小写，资源名称使用复数。
- 不跨模块注册路由，不修改无关路由。
- 新增接口必须按现有方式注册认证中间件，并核对是否还需要业务授权。
- Path、Query、Body、Header 的命名和类型必须与前端及文档一致。

## Controller 与响应

- 使用项目现有参数绑定和校验方式。
- 统一返回 `models.NewSuccessResponse()` 和 `models.NewErrorResponse()`。
- 保持 `code`、`data`、`error`、`message` 四个字段，不手写另一套 JSON 结构。
- 参数错误、未认证、无权限、未找到、冲突和内部错误应使用稳定、合适的 HTTP 状态码。
- 不把 SQL、文件路径、堆栈或其他内部实现细节直接返回给客户端。
- 内部错误记录必要上下文，对外返回稳定且符合项目风格的提示。

## Service

- 一个 Service 方法完成一个边界清晰的业务操作。
- CRUD、数据组装、批量处理、导入导出和数据权限逻辑统一放在 Service。
- 优先复用已有 Service、公共查询和转换函数。
- 避免超长函数；拆分时按业务职责拆分，不制造只有一行转发的无意义层次。
- 业务错误应保留可判断的类型或语义，不依赖字符串匹配判断错误类别。

## Model 与字段

- 数据库 Model、请求 DTO、响应 DTO 按项目现有方式放入 `models`。
- Go 字段、JSON Tag、数据库字段和前端字段保持一致。
- 接口响应字段统一使用小驼峰 JSON 命名。
- 对用途不同、校验规则不同的结构使用明确 DTO，不直接暴露不应返回的数据库字段。
- 避免重复结构体；局部临时结构确有必要时保持范围最小。

## 数据库与 GORM

- 数据库使用 MySQL，ORM 统一使用 GORM。
- 主键沿用带横线的 UUID，新增数据使用项目已有 UUID 生成方式。
- UUID Path、Query 和 Body 参数必须按现有方式校验，不能把非法值直接传入查询。
- 优先使用 `Where`、`Scopes`、`Select`、`Order`、`Limit`、`Joins` 和 `Preload` 等现有写法。
- 允许使用 GORM 占位符参数化条件，例如 `Where("user_id = ?", userID)`。
- 禁止使用 `fmt.Sprintf` 或字符串拼接把用户输入写入 SQL。
- 动态排序字段必须经过白名单映射，排序方向只允许 `asc`、`desc`，不得直接拼接请求参数。
- 检查所有数据库调用返回的 `Error`，不得忽略关键查询或写入错误。

## 查询与性能

- 分页参数统一使用 `page`、`pageSize`，排序参数使用 `sorts`，多选参数沿用逗号分隔格式。
- 分页默认值、最小值和最大值沿用现有实现；新增入口不得允许无上限的大页查询。
- 列表接口避免 N+1、循环查询和重复查询，优先批量查询、关联查询或合理的 `Preload`。
- 只查询需要的字段，大数据列表必须分页。
- 新增复杂或高频查询时检查索引使用情况，必要时使用 `EXPLAIN`，但不直接操作生产库。

## 事务

以下场景应使用事务：

- 多条关联写操作必须整体成功或失败。
- 多表写入或批量写入要求原子性。
- 读后写过程要求并发一致性。
- 任一步失败都不能保留部分结果。

单条原子 `INSERT`、`UPDATE`、`DELETE` 不强制开启显式事务。

事务中必须始终使用传入的 `tx`，不得混用全局 `database.DB`。所有错误路径都要正确回滚；外部网络调用和耗时文件操作原则上不要放在数据库事务内。

## 数据库结构变更

- 数据库设计有疑问时先询问，不自行增加重复字段或改变已有字段语义。
- 已执行或已发布的 SQL 脚本不直接修改，应新增迁移脚本。
- 表、字段、类型、索引和批量数据变更必须说明兼容性、数据风险及回滚方案。
- 未经明确授权不执行迁移、不批量修改数据、不使用自动迁移代替正式迁移脚本。
- 前后端类型、Model、查询和 Swagger 必须与数据库变更同步。

## 时间

- 当前业务接口默认时间格式为 Go layout：`2006-01-02 15:04:05`。
- 解析本地业务时间时使用 `time.ParseInLocation` 和项目统一的 `*time.Location`。
- 不依赖部署机器隐式时区新增时间逻辑；存储、比较和输出必须使用一致时区。
- 外部协议明确要求 UTC 或 RFC3339 时，以接口契约为准并在边界处转换。
- 不在不同模块重复实现时间解析和格式化函数。

## 认证与授权

- 认证统一使用 `Authorization: Bearer <token>` 和现有 JWT 中间件。
- 不新增 Cookie 登录、自定义认证 Header 或 Controller 内 Token 解析。
- 认证只表示用户身份有效，不能代替接口授权和数据权限校验。
- 授权沿用项目已有数据库、角色、权限码和业务校验方式；当前没有的统一权限能力不能假设已经存在。
- 不按用户名、角色显示名称或 `superAdmin` 字符串写死管理员权限。
- 修改 JWT、登录、认证中间件或权限逻辑前必须说明影响和兼容性。

## 文件上传下载

- 复用现有上传目录、文件 Service 和接口，不新增另一套上传逻辑。
- 校验文件名、大小、类型和路径，防止路径穿越和任意文件覆盖。
- 数据库写入与文件落盘要考虑失败清理，避免孤儿记录或孤儿文件。
- 不把本地绝对路径或内部错误直接返回客户端。

## Swagger 文档

本项目已经启用 Swagger。新增或修改接口时必须：

- 同步维护 Controller 中的 Swagger 注释。
- 使用项目既有方式生成并核对 `docs/docs.go`、`docs/swagger.json`、`docs/swagger.yaml`。

`docs` 下文件是生成文件，不手工编辑。不要执行会通过 `@latest` 隐式升级 Swagger 工具的命令，除非用户明确同意。

启动程序会尝试生成 Swagger，并向 Apifox 发起外部同步请求。因此未经明确授权，不执行 `go run .`、启动服务或其他会触发 Apifox 同步的操作。

## 错误与日志

- 所有 `error` 必须及时处理或明确向上返回，不得忽略、吞掉或用空分支处理。
- 不使用 `panic` 处理普通业务错误。
- 使用项目现有日志方式，不保留 `fmt.Println` 等临时调试输出。
- 日志不得记录密码、Token、完整 Authorization Header、密钥或敏感连接信息。
- 日志应包含定位问题所需的业务上下文，但避免输出完整请求体中的敏感字段。

## Go 编码

- 修改的 Go 文件必须使用 `gofmt`，但不要格式化无关文件。
- import、命名、函数风格和注释保持现有代码一致。
- 函数职责清晰，避免重复代码、超长函数和无意义抽象。
- 不新增依赖或升级 Go 模块，除非需求明确授权。
- 不修改公共模型、统一响应、配置加载、GORM 初始化或公共工具行为，除非需求明确要求并已评估调用方。

## 验证

在 `D:\vben\hive-admin-go` 中按修改范围执行：

```bash
gofmt -w <本次修改的.go文件>
go test ./...
go vet ./...
go build ./...
```

当前仓库没有 `_test.go` 文件，`go test ./...` 主要用于编译检查。新增可独立验证的业务逻辑或修复重要 Bug 时，应补充针对性测试；没有实际测试用例时不得声称完整测试已通过。

不要为了验证启动服务、连接生产数据库、执行迁移或同步 Apifox。命令因环境或存量问题失败时，应说明失败位置及其与当前修改的关系。

## 完成检查

- 是否保持 Router、Controller、Service、Model 分层。
- 是否复用了已有实现并保持统一响应。
- 是否正确处理参数、错误、事务、权限和敏感信息。
- 是否存在 SQL 注入、N+1、循环查询或无上限查询。
- 是否同步 Swagger、前端 API 和数据库脚本。
- 是否执行了必要验证，是否影响其它模块和接口。

最终代码应保持一致的架构、命名、接口、数据库和错误处理风格，优先稳定性、正确性和可维护性。
